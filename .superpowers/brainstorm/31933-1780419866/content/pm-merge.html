修复：PM Agent 负责合并 + 关闭 Issue

Coder/Tester/Reviewer 只提供信号。PM Agent 是唯一可以 merge + close 的角色

🎭 角色权限矩阵

操作 PM Coder Tester Reviewer
创建 Issue✅❌❌❌
写 .status/*-dev✅✅❌❌
写 .status/*-test❌❌✅❌
写 .status/*-review❌❌❌✅

🔄 简化后的 Agent 循环

💻 Coder

while true:
  读黑板

  // 找新任务
  for F 拓扑序:
    if 🔴 && 前序全🟢:
      mkdir F-lock  // 原子
      失败→continue
      echo 🟡 > F-dev
      rmdir F-lock
      编码→自测→push
      pr=$(gh pr create)
      echo $pr > F-pr
      echo 🟣 > F-dev
      break

  // 修复被退回的
  for F 拓扑序:
    if 🟡 && (test❌ || review❌):
      retry=$(cat F-retry)
      if retry>=3:
        echo ⚠️ > F-dev
        continue
      修复→push
      echo 🟣 > F-dev
      echo ⏳ > F-test
      echo ⏳ > F-review
      break

  // Coder 不做合并检查
  sleep 5min

🧪 Tester

while true:
  读黑板

  for F:
    if 🟣 && test=⏳:
      生成验收测试
      mvn test
      if 通过: echo ✅ > F-test
      else:    echo ❌ > F-test
      break

  // Tester 不做合并
  sleep 5min

🔍 Reviewer

while true:
  读黑板

  for F:
    if 🟣 && review=⏳:
      4维审查
      if 通过:
        echo ✅ > F-review
        gh pr approve $pr
      else:
        echo ❌ > F-review
        gh pr request-changes $pr
      break

  // Reviewer 不做合并
  sleep 5min

🤖 PM（唯一合并者）

while true:
  读黑板

  for F 拓扑序:
    if 🟣 && test✅ && review✅:
      // 全部通过 → 合并！
      gh pr merge $(cat F-pr)
      gh issue close ${F#F}
      更新 feature-map README
      echo 🟢 > F-dev
      break

    if 🟣 && (test❌ || review❌):
      // 有人不通过 → 退回
      retry=$(cat F-retry)
      echo $((retry+1)) > F-retry
      if $(cat F-retry) >= 3:
        echo ⚠️ > F-dev
      else:
        echo 🟡 > F-dev
      break

  // 所有🟢？退出
  if 全部🟢:
    exit 0

  sleep 5min

✅ 为什么 PM 做合并是正确的

1. 职责闭环PM 创建 Issue → PM 关闭 Issue。Issue 生命周期由同一个人管理
2. 决策单一Coder/Tester/Reviewer 只提供信号，不做决策。不会出现"两个 Agent 同时 merge"
3. 晚上自动化PM Agent 在全自动模式下不需要人类确认——test✅ + review✅ 就是合并条件（你在 PM 对话阶段已确认过需求）
4. 白天人工模式PM Agent 也可以设置"关键 Feature（P0）合并前需人工确认"，白天工作时可以加这个检查点
