Agent 通信协议：状态黑板（无中心调度器）

每个 Agent 只做三件事：读黑板 → 干活 → 写黑板。黑板是唯一的真相来源。

🤖 每个 Agent 的执行循环

💻 Coder（自动循环）

while true:
  1. 读 feature-map/README.md
  2. 找第一个 🔴且前序全🟢 的 Feature
  3. 没有 → sleep 5min，continue
  4. 更新状态 🔴→🟡
  5. 读 Feature 文件 + Upstream Context
  6. 编码（按规范）
  7. git push + gh pr create
  8. 更新状态 🟡→🟣 + 填 PR 号

🔍 Reviewer + 🧪 Tester（监听 PR）

while true:
  1. 读 feature-map/README.md
  2. 找 🟣待审查 的 Feature
  3. 没有 → sleep 5min，continue
  4. Tester: 生成测试 → 执行
  5. Reviewer: 4 维规范审查
  6. 任一失败 →
      PR 上 comment "❌ 原因"
      状态 🟣→ 🟡（回 Coder）
  7. 全部通过 →
      PR 上 comment "✅ LGTM"
      gh pr merge
      状态 🟣→ 🟢
      gh issue close

🔑 状态表协议（唯一需要标准化的东西）

# feature-map/README.md 的状态索引表 —— 唯一黑板
| ID | Feature | 前序 | 状态 | PR | 尝试 | |----|---------|------|------|----|------| | F1 | 用户注册 | - | 🟢已完成 | #11(merged) | - | | F3 | 用户登录 | F1,F2 | 🟣待审查 | #13(open) | - | | F4 | 密码重置 | F1,F2 | 🟡开发中 | - | - | | F5 | 会话管理 | F3 | 🔴待开发 | - | - | | F6 | 登录保护 | F3 | 🔴待开发 | - | - |
# 协议规则： # 1. 状态列是单写者：只有当前工作在这个 Feature 上的 Agent 可以改 # 2. Agent 认领 Feature 时必须立即更新状态（🔴→🟡 或 🟣→🟡） # 3. "尝试"列记录重试次数，防止死循环（审查失败 >3 次，标记 ⚠️需人工介入） # 4. Coder 不能认领前序未完成（前序列不是全🟢）的 Feature # 5. 并行 Coder 可以同时认领多个前序全🟢 的 Feature（如 F4,F5,F6 都依赖 F3） # 但如果一个 Coder 在跑 F3（🟡），F4/F5/F6 的前序还不是全🟢，不能认领
