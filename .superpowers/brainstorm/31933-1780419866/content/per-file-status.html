方案 A 深入：每人一个文件 + 定时合并检查

消除两个竞态条件：写文件冲突 + "谁最后写完谁来合并"的不确定性

📂 文件结构：一个文件一个人写

feature-map/
├── README.md ← 人类可读的全局 DAG（PM 生成，后续只读）
├── .status/ ← Agent 通信目录（每个文件只写 1 个值）
│ ├── F1-dev ← 内容: 🟢 Coder 写
│ ├── F1-test ← 内容: ✅ Tester 写
│ ├── F1-review ← 内容: ✅ Reviewer 写
│ ├── F1-pr ← 内容: #11 Coder 写
│ ├── F1-retry ← 内容: 0 任何人写
│ ├── F2-dev
│ ├── F2-test
│ └── ...
├── F1-用户注册.md ← Feature 上下文（PM 生成，只读）
└── F2-邮箱验证.md

🔑 关键设计：每个 Agent 循环结束时都检查"合并条件"

不需要"最后写完的 Agent 触发合并"，而是任何 Agent 在每次循环中检查所有 🟣Feature。解决了"两人同时写完都以为自己是第一个"的问题。

💻 Coder 循环

while true:
  读黑板(cat .status/*)

  // 步骤1：找新任务
  for F in 按拓扑序:
    if F-dev=="🔴" && 前序全🟢:
      echo "🟡" > F-dev
      读Feature文件+Upstream
      编码→自测→git push
      pr=$(gh pr create)
      echo $pr > F-pr
      echo "🟣" > F-dev
      goto 步骤3

  // 步骤2：修复被退回的
  for F in 按拓扑序:
    if F-dev=="🟡" && (F-test=="❌" || F-review=="❌"):
      retry=$(cat F-retry)
      if retry >= 3: echo "⚠️" > F-dev; continue
      读PR comment→修复→push
      echo "🟣" > F-dev
      echo "⏳" > F-test
      echo "⏳" > F-review
      goto 步骤3

  // 无工作
  sleep 5min; continue

  // 步骤3：检查合并条件（关键！）
  for F in 所有Feature:
    if F-dev=="🟣" && F-test=="✅" && F-review=="✅":
      gh pr merge $(cat F-pr) --squash
      gh issue close ${F#F}
      echo "🟢" > F-dev
    elif F-dev=="🟣" && (F-test=="❌" || F-review=="❌"):
      retry=$(cat F-retry)
      echo $((retry+1)) > F-retry
      if $(cat F-retry) >= 3:
        gh pr comment $(cat F-pr) "⚠️ 重试>3次，需人工介入"
        echo "⚠️" > F-dev
      else:
        echo "🟡" > F-dev

🎯 为什么这个设计消除了竞态条件

竞态场景为什么被消除
Tester 和 Reviewer 同时写状态文件 不存在——Tester 只写 F{n}-test，Reviewer 只写 F{n}-review，不同文件
"谁最后写完谁决定合并" 不存在——每个 Agent 循环结束时都检查。即使两人同时写完、同时检查，它们读到的 test + review 文件内容是一样的，得出的合并/退回结论也一样。最坏情况：两人同时 merge → 第二次 merge 失败（已合并），报错而非产生错误状态
Coder 修复后与 Tester 同时写 不存在——Coder 写 F{n}-dev 和 F{n}-test/F{n}-review，Tester 读-处理-写。Coder 写入后，Tester 在下一次循环才读到新状态
两个 Coder 同时认领同一个 Feature ⚠️ 仍可能——需要额外保护。方案：认领前除了读 🔴，还要检查 F{n}-dev 的文件修改时间。如果最近 10 秒内被改过（可能另一个 Coder 刚认领），等待重试

⚠️ 两个 Coder 同时认领同一个 Feature 的解决

# Coder 认领逻辑（带时间戳保护）
读黑板 → F3-dev 内容为 "🔴"
检查 F3-dev 的 mtime: stat -f %m .status/F3-dev → 1780419900 now = date +%s → 1780419905 差值 5 秒 < 10 秒 → ⚠️ 可能被刚认领，跳过 F3，等下一轮
# 为什么 10 秒就够了？ # Agent 循环先"读"再"claim"再"写"，周期内不会出现 2 个 Agent 同时读到相同的 🔴 # 如果真的刚好同时（概率极低），mtime 差值为 0-1s，保护机制触发
# 更稳妥的做法：不用时间戳，用 mkdir 原子操作 # mkdir .status/F3-lock → 成功→ 归我 / 失败→ 别人已认领 # 文件系统的 mkdir 是原子操作，不可能两个同时成功

📐 完整协议总结

规则内容
1. 单写者每个文件只有 1 个 Agent 角色可以写。F{n}-dev=Coder，F{n}-test=Tester，F{n}-review=Reviewer
2. 原子认领Coder 认领 Feature 时必须用 mkdir .status/F{n}-lock（原子操作），成功才认领
3. 周期性读每个 Agent 循环开始时 cat .status/* 读全量黑板，不缓存
4. 合并检查每个 Agent 循环结束时检查所有 🟣Feature 的 test+review，合并或退回
5. 重试上限退回 >3 次 → ⚠️需人工介入，不无限循环
6. 锁清理认领完成后 rmdir .status/F{n}-lock。如果 Agent 崩溃，.status/F{n}-dev 保持 🟡，超时（30分钟）后其他 Agent 可强行清理锁
