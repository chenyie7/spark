最干净的方案：GitHub 做通信层，PM 是唯一写文件的人

本地文件零竞争。Coder/Tester/Reviewer 只操作 GitHub，PM 读 GitHub 做决策，写本地文件做记录

🎯 核心原则

只有 PM 能改本地文件。Coder/Tester/Reviewer 只操作 GitHub。
通信靠 GitHub PR + Checks + Reviews，不靠本地文件

📡 "通知"机制：GitHub 原生，不需要自己造

Coder
通知方式：
gh pr create
→ PR #13 出现在 GitHub 上
Tester
通知方式：
gh pr comment "✅"
→ PR #13 出现评论
Reviewer
通知方式：
gh pr review --approve
→ PR #13 状态变为 "Approved"
↓
PM Agent 轮询 GitHub（每 1 分钟）

🤖 PM Agent 的判断逻辑

# PM Agent 循环 — 所有信息来自 GitHub
while true:
# 1. 读 GitHub（唯一真相源） prs = gh pr list --json number,title,body,reviews,comments
for pr in prs: # 从 PR title 提取 Feature ID： "F3: 用户登录" → F3 fid = extract_feature_id(pr.title)
# 2. 判断条件 test_ok = any("[✅] Tests passed" in c.body for c in pr.comments) review_ok = any(r.state == "APPROVED" for r in pr.reviews)
if test_ok and review_ok: # 3. 合并 + 关闭 gh pr merge {pr.number} --squash --auto gh issue close {issue_number} # 4. 写本地文件（只有 PM） 更新 feature-map/README.md: {fid} → 🟢
elif not test_ok and not review_ok: # 有人不通过 → 不操作，等 Coder 修复后重新提 PR
sleep 60 # 1 分钟轮询

维度方案 A（本地文件多人写）方案 B（GitHub 通信 + PM 唯一写）
本地文件竞争 有 — 需要 lock/mkdir 无 — 只有 PM 写本地文件

通知机制 自己实现 GitHub 原生：PR review/comments/checks
