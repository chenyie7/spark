🤖 Coder Agent 的完整执行流程

# ===== Coder 循环 =====
while true:
# Step 1: 找任务 — 读 PM 维护的唯一本地文件 dag = read feature-map/README.md for feature in topological_order(dag): if feature.status == "🔴" and all_predecessors_are("🟢"): target = feature break
if target is None: # 没有可开发的任务 — 检查是否有被退回的 for feature in topological_order(dag): if feature.status == "🟡": # 被退回，需要修复 pr = gh pr view --json number,comments,reviews #{feature.pr} if pr.reviews has "CHANGES_REQUESTED": target = feature # 需要修复 break
if target is None: sleep 300; continue
# Step 2: 开发 git checkout -b feature/F{target.id} read feature-map/F{target.id}-{name}.md read upstream_context from feature file read agents/coder/ 规范文件 implement code run self-test: mvn test
# Step 3: 提交 PR git add . && git commit -m "feat: F{id} {name}" git push -u origin feature/F{id} pr_num = gh pr create \ --title "F{id}: {name}" \ --body "关联 Issue #{id} | Feature: [F{id}](feature-map/F{id}-{name}.md)" \ --base main
# Step 4: 回填 Implementation Layer（Feature 文件，不是状态文件） update feature-map/F{id}-{name}.md: Implementation Layer: Changed Files: [list] Exported APIs: [list] Exported Models: [list] git add feature-map/F{id}-{name}.md && git commit && git push
# Coder 不写 README.md 状态表！那个只有 PM 写 # Coder 不写 .status/ 文件！没有 .status/ 目录

🧪 Tester Agent 的完整执行流程

# ===== Tester 循环 =====
while true:
# Step 1: 找待测试的 PR open_prs = gh pr list --state open --json number,title,labels,comments
for pr in open_prs: # 判断是否已经被我测试过 already_tested = any("🧪 Test Result" in c.body for c in pr.comments if c.author == "coder-bot") if not already_tested: target = pr break
if target is None: sleep 60; continue
# Step 2: 生成并运行验收测试 fid = extract_feature_id(pr.title) # "F3: 用户登录" → F3 ac = read PRD.md section for F{fid} git checkout feature/F{fid} generate acceptance tests from ac.Given/When/Then result = mvn test
# Step 3: 通过 PR 评论通知 PM if result.passed: gh pr comment {pr.number} --body "## 🧪 Test Result ✅ All {result.total} acceptance tests passed {result.summary}" else: gh pr comment {pr.number} --body "## 🧪 Test Result ❌ {result.failed}/{result.total} tests failed {result.failures}"
git checkout main && git branch -d feature/F{fid}
# Tester 不写任何本地文件 # Tester 不 merge、不 close issue

🔍 Reviewer Agent 的完整执行流程

# ===== Reviewer 循环 =====
while true:
# Step 1: 找待审查的 PR open_prs = gh pr list --state open --json number,title,reviews
for pr in open_prs: my_review = [r for r in pr.reviews if r.author == "reviewer-bot"] if not my_review: # 还没审查过 target = pr break
if target is None: sleep 60; continue
# Step 2: 审查代码 fid = extract_feature_id(pr.title) git checkout feature/F{fid} read agents/reviewer/structure-check.md read agents/reviewer/quality-check.md read agents/reviewer/auth-check.md read agents/reviewer/infra-check.md result = run_4_dimension_review()
# Step 3: 通过 PR Review 通知 PM if result.all_pass(): gh pr review {pr.number} --approve --body "✅ LGTM" else: gh pr review {pr.number} --request-changes --body "{result.issues}"
git checkout main && git branch -d feature/F{fid}
# Reviewer 不写任何本地文件 # Reviewer 不 merge、不 close issue

🤖 PM Agent 的完整执行流程（唯一决策者）

# ===== PM 循环 =====
while true:
# Step 1: 从 GitHub 读取所有 PR 状态 open_prs = gh pr list --state open --json number,title,reviews,comments
for pr in open_prs: fid = extract_feature_id(pr.title)
# Step 2: 判断测试和审查是否都完成了 test_comment = find_comment_by_pattern(pr.comments, "🧪 Test Result") review = get_latest_review_by_me_or_bot(pr.reviews)
# Step 3: 决策 if test_comment and "✅" in test_comment.body: test_ok = True elif test_comment and "❌" in test_comment.body: test_ok = False else: continue # 还在等 Tester
if review and review.state == "APPROVED": review_ok = True elif review and review.state == "CHANGES_REQUESTED": review_ok = False else: continue # 还在等 Reviewer
if test_ok and review_ok: # ✅ 合并！ gh pr merge {pr.number} --squash --delete-branch gh issue close {fid} gh issue comment {fid} --body "✅ Merged via PR #{pr.number}" # 更新本地状态表（只有 PM 写） update feature-map/README.md: F{fid} → 🟢 git add feature-map/README.md && git commit && git push
elif not test_ok or not review_ok: # ❌ 不合并，不操作。等 Coder 修复后重新提 PR # Tester 和 Reviewer 的评论已经说明了原因 # Coder 读到 CHANGES_REQUESTED 或 ❌ comment 就知道要修复
# Step 4: 所有 Issue 都关了？ all_done = all issues from PM's feature-map are closed on GitHub if all_done: print "🌅 全部完成！{n}/{n} Issues 已关闭。" exit 0
sleep 60

🔄 Gitea 能否实现？

功能 GitHub CLI Gitea CLI (tea) 可用？
创建 PR gh pr create --title --body --base tea pr create --title --description --base ✅
列出 PR gh pr list --state open --json tea pr list --state open --output json ✅
PR 评论 gh pr comment {n} --body tea comment --repo xxx --type pr ✅
PR Review (approve) gh pr review --approve tea pr review --approve ✅（Gitea 1.18+）
PR Review (request changes) gh pr review --request-changes tea pr review --request-changes ✅
PR 合并 gh pr merge --squash tea pr merge --squash ⚠️ 需要 Gitea 1.20+
关闭 Issue gh issue close tea issue close ✅
JSON 输出解析 gh pr list --json --jq tea pr list --output json | jq ✅
PR Checks/CI 状态 gh pr checks REST API /repos/{owner}/{repo}/statuses ✅

🎯 Gitea 结论：完全可以，只需换 CLI

Gitea 版本要求≥ 1.20（2024年发布的版本，支持 PR review + merge）
CLI 工具tea（Gitea 官方 CLI）替代 gh
差异点tea 命令参数名略有不同（--description 代替 --body），JSON 输出字段名略有差异
适配成本Agent prompt 中替换 CLI 命令即可，架构不需要改
私有部署优势无需外网、数据自控、审阅速度不受 GitHub API 限流影响
