PM Agent 的核心难题

🤔 难题 1：需求是不确定的

和真人 PM 对接时，需求是模糊的、会变的、需要多轮确认的。Agent 怎么"聊"出靠谱的需求？

📝 难题 2：输出物定义

PM Agent 到底产出什么？PRD 文档？技术规格？API 设计？数据库设计？还是全部？

🔄 难题 3：边界在哪

PM 和 Analyst 是一个 Agent 还是两个？需求确认到什么程度算"够了"？谁来拍板？

🔗 难题 4：如何传递给下游

PM 的输出怎么给 Coder 用？Issue 格式？Markdown 文档？结构化 JSON？如何保证信息不丢失？

我们先聚焦第一个问题

PM Agent 的输入是什么？它和谁交互？

A

和人类 PM 对接

人类 PM 给出模糊需求 → PM Agent 结构化 → 输出 PRD。Agent 是"需求翻译官"

B

直接和业务方/用户对接

跳过人类 PM，Agent 直接对话业务方，自己就是 PM

C

人类 PM 已经把需求写好了

Agent 只做"需求 → PRD → 技术设计"的结构化转换，不做需求挖掘

D

混合模式

支持以上多种入口，根据场景灵活切换
