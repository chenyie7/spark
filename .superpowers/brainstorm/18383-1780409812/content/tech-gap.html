PM 只出 PRD+Issues 之后，技术方向谁来管？

方案 A 的链路：PM(人) → PM Agent → PRD + Issues → Coder Agent。中间有个缺口

问题：没有技术约束时可能发生什么

Issue 1
用户登录
→
Coder 用了
Session 方式
vs
Issue 2
订单查询
→
Coder 用了
JWT 方式

❌ 两个 Issue 分别实现，认证方案不一致，代码合不到一起

三种补位方案

A

PM Agent 多输出一份"技术概要"

PRD 之后、Issues 之前，加一个步骤：由 PM Agent 根据项目规范，输出一份轻量级技术方向说明（认证方案、数据库选型、包结构约定）。Issues 中引用它。不另建 Agent

B

Coder Agent 开发前先做技术规划

coder 读取 Issue 后、写代码前，先输出一份技术设计（选什么方案、怎么实现），让人类或 PM 确认后再动手。不建新 Agent，但增加一个审查点

C

现在就建 Analyst Agent（轻量版）

不用等到 C 方案。先做一个最小化的 Analyst：读取 PRD + 项目规范 → 输出技术决策单（架构方案、接口规范、数据库变更）。PM 和 Coder 之间的一道"翻译层"
