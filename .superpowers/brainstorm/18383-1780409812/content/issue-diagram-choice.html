Issue 里放 Mermaid 图还是放链接？

从"AI 大概率会读取并理解"的角度分析

两种方案对比

方案 A：Issue 里嵌入 Mermaid 代码
## 依赖关系总览

```mermaid
graph TD
  F1[F1: 用户注册] --> F2[F2: 邮箱验证]
  F1 --> F3[F3: 用户登录]
  ...
  F7 --> F8[F8: 加入购物车]
  F8 --> F9[F9: 下单结算]
  F7 --> F10[F10: 商品收藏]
```

## 状态流转

```mermaid
stateDiagram-v2
    [*] --> pending: 用户注册
    pending --> active: 邮箱验证通过
    ...
```

❌ Issue Body 过长，图表多了很混乱
❌ AI 读 Mermaid 代码 = 读一堆箭头和括号，信息密度低
❌ DAG 变了要改 N 个 Issue，容易不一致
✅ 不需要额外跳转

方案 B：结构化摘要 + 文件链接
## 依赖关系

```
上游依赖（必须先完成）：
  F3: 用户登录
  F7: 商品浏览

下游被依赖（本节点完成后解锁）：
  F9: 下单结算
```

## 状态流转

用户状态：pending → active → locked → deleted
关键规则：locked 返回 423，不提示"密码错误"

> 📊 完整 DAG 和状态机见 [F8 Feature Map](link)

✅ Issue Body 精简，AI 直接读到关键依赖
✅ 结构化文本比 Mermaid 代码信息密度高
✅ DAG 变了只改源文件，Issues 不过期
✅ AI 读完摘要后，需要详情时跟随链接

AI 读哪种更高效？实测视角

AI 需要知道什么 Mermaid 代码给了什么 结构化摘要给了什么
哪些依赖必须先完成？ 需要解析 F3 --> F8 语法 Depends On: F3, F7 直接可读
用户有哪些状态？ 从 pending --> active: xxx 推断 pending → active → locked → deleted 一目了然
locked 状态返回什么错误码？ 在 note 里翻找 locked → 423，不提示"密码错误"
前序节点可复用哪些代码？ Mermaid 图不包含这个信息 Upstream Context 表格直接列出

推荐方案：Issue 用结构化摘要 + 文件链接

为什么这样更好？

1. AI 解析成本"Depends On: F3, F7" 比解析 Mermaid 图语法快得多，AI 直接拿到结构化信息
2. Coder Agent 的流程Coder Agent 的 prompt 会明确写"读取 feature-map/F{n}.md"，Agent 一定会读源文件。Issue 不需要重复放图
3. 单一数据源DAG 图在 feature-map/ 里是源，改了不改 Issues。不担心过期不一致
4. GitHub 渲染Mermaid 图在 feature-map 文件里照样渲染。人类想看图时点链接即可，不损失可视化
