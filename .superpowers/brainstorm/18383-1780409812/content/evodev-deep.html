EvoDev 深度分析：到底哪些值得我们实现？

来源：arXiv 2511.02399，Fudan + NTU，2025.11。在 Android 开发任务上 超出 Claude Code 56.8%

📊 EvoDev 完整架构（3 阶段 6 角色）

Stage 1: 整体设计
📋 Business Analyst
结构化需求
🏗️ Architect
UI设计 + 数据实体
Stage 2: Feature Map
🔍 Feature Extractor
需求 → 特性列表
📐 Feature Planner
依赖分析 → DAG
Stage 3: 迭代开发
👔 Chief Programmer
拓扑分配 + 收集上下文
💻 Programmer
写代码 + 测试

🔑 DAG 的核心：不是"依赖图"，是"上下文传播管道"

// Feature Map DAG 的本质
F3: 商品详情 ← 完成开发后，节点携带：
business_layer: 展示商品图片、价格、规格、库存
design_layer: ProductDetailActivity, ProductViewModel,
product_detail.xml, ProductRepository
impl_layer: 改动的文件: Product.java(+85行),
ProductViewModel.kt(+120行), build.gradle(+3行)

↓ context 沿边向下传播 ↓

F4: 加入购物车 ← 开发时收到的上下文：
upstream_business: F3 展示商品后，用户可点击"加入购物车"
upstream_design: 复用 ProductViewModel, ProductRepository
新增 CartActivity, CartViewModel
upstream_code: 可引用: com.example.model.Product（F3 已定义）
可调用: ProductRepository.getById()（F3 已实现）
用户会话: 沿用 F1 的 SessionManager

✅ 这样 F3 完成后，F4 的开发 Agent 知道：
该复用什么、该新增什么、接口契约是什么

⚖️ 与我们的设计逐项对比

维度 EvoDev 我们的设计 判断
需求结构化 Business Analyst + Architect（2 角色） PM Agent 对话 + 模块化 PRD 模板 ✅ 功能等价，不分拆
特性拆解 Feature Extractor + Feature Planner（2 角色） PM Agent Issue 拆分（含依赖关系） ✅ 功能等价
节点上下文丰富度 3 层：business + design + code
每层详细、可被下游消费		 Issue 里放 PRD 引用 + 技术提示
缺少 code_context 层		 ⚠️ 关键差距
上下文传播 DAG 边自动传播前序节点的
完整上下文到后续节点		 仅标记依赖关系，#3 depends on #1
不传递具体上下文		 ⚠️ 关键差距
任务分配 Chief Programmer 拓扑排序分配 Coder 按依赖顺序认领 Issue ✅ 等价
代码生成 Programmer（含测试） Coder Agent + 技术规划阶段 ✅ 等价
代码审查 无独立 Review 角色 Reviewer Agent（4 维审计） ✅ 我们的增量优势

🎯 结论：DAG 部分值得实现，但不需要照搬全部

A

轻量 DAG：增强 Issue 节点 + Coder 收集上下文

PM Agent 侧：Issue 模板增加 business_layer + design_layer 字段
Coder 侧：开发前自动读取依赖 Issue 的实现产物（已改动的文件列表、新增的 API/Model），作为上下文注入
不建独立的 Feature Map DAG 文件，利用 Issue 的依赖关系动态构建

改动量：中。复用现有 Issue 体系
B

完整 DAG：独立 Feature Map 文件

PM Agent 额外输出一个 feature-map.yml，包含完整的 DAG 结构 + 每个节点的 3 层上下文。
Coder 从 Feature Map 而非 Issue 获取上下文。
和 EvoDev 完全对齐

改动量：大。新增概念和文件格式
