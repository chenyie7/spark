我们的设计 vs EvoDev 对比

EvoDev 的核心创新不是"又多了几个角色"，而是上下文的传递方式

角色映射

EvoDev 角色 我们的设计 差距
Feature Extractor PM Agent（对话收集需求） ✅ 功能重叠
Feature Planner
→ DAG + business_logic + design_spec + code_context		 PM Agent（Issue 拆分 + 依赖关系） ⚠️ 我们只有"依赖关系"，缺少每个节点携带的 rich context
Chief Programmer
→ 拓扑排序 + 收集前序上下文		 Coder Agent（技术规划阶段） ⚠️ 我们没有"前序节点代码上下文收集"这个步骤
Programmer Coder Agent（写代码阶段） ✅ 功能重叠
— Reviewer Agent ✅ 我们的增量优势

EvoDev 的真正精髓：Feature Node 携带的上下文

F4: 加入购物车
depends_on: ["F3:商品详情"]

business_logic: |
用户从商品详情页点击"加入购物车"按钮...

design_spec: |
API: POST /cart/items { product_id, quantity }
Response: { cart_id, items[], total_price }

code_context: |
// F3 中已定义的 Product 组件和数据模型
// 购物车使用 F1 中定义的用户会话管理

关键在于 code_context：后一个 feature 的开发 Agent 知道自己可以用前序节点已实现的哪些代码、模型、组件。这避免了每个 Issue 从头开始、各自为政

能不能实现？怎么融合？

A

轻量融合：增强 Issue 模板

不改 PM Agent 架构，在 Issue 模板中增加 business_logic + design_spec 字段。
code_context 由 Coder Agent 在开发时从已完成 Issue 中动态收集

改动量最小，立即可做
B

完整融合：PM Agent 输出 Feature DAG

PM Agent 不只出 Issues，而是输出一个完整的 FeatureMap（DAG + 每个节点的 rich context）。
Coder Agent 按拓扑序开发，获得前序节点的完整上下文

改动量大，但和 EvoDev 完全对齐
C

分步演进（推荐）

第一步（PM Agent v1）：按当前设计实现，Issue 模板增加 business_logic + design_spec
第二步（PM Agent v2）：引入 Feature DAG 和 code_context 传递机制
两阶段之间用真实使用反馈来验证方向

务实，不锁死方向
