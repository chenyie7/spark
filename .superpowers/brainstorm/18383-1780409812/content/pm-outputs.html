PM Agent 应该产出什么？

输入明确了（人类 PM 的模糊需求），那输出物有哪些？这决定了 Agent 的复杂度

候选输出物清单

以下是一份需求到开发全链路中可能需要的文档，PM Agent 负责哪些？

📋 PRD（产品需求文档）

用户故事、功能列表、验收标准、优先级。面向产品和开发两端

难度：中

🔧 技术规格

从 PRD 推导技术方案：接口契约、数据模型、非功能需求

难度：中高

🔗 API 设计

RESTful 接口定义：URL、Method、Request/Response、错误码

难度：中

🗄️ 数据库设计

表结构、索引、关系、迁移脚本

难度：中

📝 GitHub Issues

可执行的开发任务，拆解到可直接开发粒度

难度：低

✅ 验收测试用例

对应验收标准的可执行测试场景

难度：中

PM Agent 负责到什么程度？

A

最小范围：只做 PRD + Issues

人类 PM 给需求 → Agent 产出 PRD 文档 + 拆解为 GitHub Issues。技术设计留给 coder

B

完整规格：PRD → 技术设计全包

Agent 从需求一路推到 API 设计 + 数据库设计，coder 拿到的是完整技术规格

C

分阶段：PM → Analyst 两个 Agent

PM Agent 只做业务需求（PRD + Issues），Analyst Agent 做技术转化（API + DB）。各司其职
