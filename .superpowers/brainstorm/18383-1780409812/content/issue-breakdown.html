PRD → Issues：怎么拆？拆多细？

Issue 是 Coder Agent 的输入，拆得太粗 coder 做不完，拆得太细碎片化

第一个问题：拆分粒度

A

按功能点拆分

一个用户故事 = 一个 Issue
例："用户手机号+验证码登录"是一个 Issue

粒度：中等，适合大多数场景
B

按接口拆分

一个 API 端点 = 一个 Issue
例："POST /api/login/send-code" 单独成 Issue

粒度：细，适合 API 密集的场景
C

按用户故事（Epic → Story）

一个大需求 = 一个 Epic，拆成 3-8 个 Story
每个 Story 是一个完整的前后端开发单元

粒度：灵活，推荐

第二个问题：Issue 里放什么信息？

1

引用 PRD 对应章节

链接到 PRD 中的用户故事和验收标准

2

验收标准内联

把 Given/When/Then 直接写在 Issue 里，coder 不需要跳转查 PRD

3

技术提示

建议涉及的包、表、接口路径前缀，给 coder 一个起点

4

依赖关系

标记 BlockedBy：Issue B 依赖 Issue A 先完成
