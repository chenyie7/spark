一个文件 vs 每个 Feature 一个文件

用你的方式更模块化，每个 Feature 可以独立开发、独立回填

❌ 原来的设计：一个大文件

feature-map.md（单文件，越来越长）
├── Mermaid DAG（全量）
├── F1 完整上下文
├── F2 完整上下文
├── F3 完整上下文
├── ...
└── F10 完整上下文

问题：多人并行开发时，Coder 回填同时改同一个文件，冲突频繁。

✅ 你的方案：每个 Feature 独立文件

feature-map/
├── README.md ← 全局 Mermaid DAG + 拓扑排序 + 文件索引
├── F1-用户注册.md ← 单 Feature 完整上下文
├── F2-邮箱验证.md ← 含 upstream_context 引用
├── F3-用户登录.md
├── F4-密码重置.md
├── F5-会话管理.md
├── F6-登录保护.md
├── F7-商品浏览.md
├── F8-加入购物车.md
├── F9-下单结算.md
└── F10-商品收藏.md

优势对比

一个大文件每个 Feature 独立文件
并行开发❌ 回填时冲突频繁✅ 各改各的，不冲突
Issue 引用链接到文件锚点✅ 直接链接到自己的文件
上下文清晰度混在一起，滚动查看✅ 打开即看，只关心自己的上下游
全貌总览✅ 一个文件看完✅ README.md 有全局 DAG + 索引表
状态追踪靠锚点定位✅ README 表格一目了然
