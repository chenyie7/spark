不是"要图还是要数据"，是"一个文件装下所有"

feature-map.md = Mermaid 图（可视化）+ Markdown 章节（结构化数据），替代 YAML

对比：两个文件 vs 一个文件

❌ 原来的想法：两个文件
# feature-map.yml（Agent 读） features: - id: F1 name: 用户注册登录 depends_on: [] business: | 用户可通过手机号+验证码... design: | API: POST /api/auth/... impl: changed_files: []

# feature-map.md（人读） ```mermaid graph TD F1 --> F2 ```
问题：两份数据要手动同步， 改了 YAML 忘了改 MD， 或者反过来。
✅ 推荐做法：一个文件
# feature-map.md（人 + Agent 都读）
## 依赖关系图 ```mermaid graph TD F1 --> F2 --> F3 F3 --> F4 ```
## 开发顺序 F1 → F2 → F3 → F4,F5 → F6
---
## F1: 用户注册登录 {#F1}
业务层：手机号+验证码注册登录 设计层：POST /api/auth/send-code POST /api/auth/login 表: user(id, phone, nickname) 实现层：AuthController.java User.java, UserService.java 上游上下文：无
---
## F4: 加入购物车 {#F4}
业务层：商品详情页点击加入购物车 设计层：POST /api/cart/items 表: cart, cart_item 复用: F3 的 ProductDTO 实现层：（待开发） 上游上下文： - models: F1→User, F3→Product, Sku - apis: F1→getLoginId(), F3→getById()
✅ 人和 Agent 看同一个文件， 信息永远一致。Markdown 的 标题就是字段名，Agent 解析容易

总结

📊 Mermaid 图可视化 DAG，在 GitHub 上直接渲染
📝 Markdown 章节每个节点完整的 business / design / impl / upstream 数据
📦 一个文件人和 Agent 读同一份，不会不同步
❌ 不需要 YAMLMarkdown 的 ## 标题 + 列表 = 天然的结构化，不需要额外格式
